# Structured Outputs via **Responses API** o que é e por que usar

* **Structured Outputs** força o modelo a **responder 100% no formato do seu JSON Schema**.
* Vantagens: **type-safe de verdade**, **detecção programática de recusas** e **prompts mais simples** (menos “não faça X, faça Y”).
* **Quando usar**:

  * **Com ferramentas (function calling)** → quando o modelo precisa **chamar** funções/sua infra.
  * **Como resposta ao usuário (text.format/json\_schema)** → quando você só quer **estruturar a saída** para renderizar UI, salvar no banco, etc.
    Regra mental: *tools → function calling*; *UI/dados → structured text.format*.
* **JSON mode ≠ Structured Outputs**: JSON mode só garante JSON válido; **não valida contra schema**. Prefira **Structured Outputs**; se o modelo/snapshot não suportar, caia para JSON mode + validação local.

# Regras essenciais do schema (que mais pegam)

* **Raiz precisa ser `type: "object"`** (nada de `anyOf` no topo).
* **Todo campo deve estar em `required`**. Se quiser “opcional”, use **`type: ["string","null"]`** (ou equivalente).
* **`additionalProperties: false` é obrigatório** em todos os objetos.
* **`anyOf` é permitido** (não na raiz) e **recursão** é suportada.
* **Limites**: até \~10 níveis de aninhamento; \~5k propriedades totais; enums e strings com limites de tamanho.
* **Ordem das chaves** no output segue a ordem do seu schema.

# Compatibilidade de modelos

* Structured Outputs via `text.format: { type: "json_schema", strict: true }` está disponível nos **modelos recentes** (ex.: 4.1/4o e gerações mais novas; GPT-5).
* **Se algum snapshot não suportar** `json_schema`, use **JSON mode** (`{ type: "json_object" }`) e valide com Zod localmente.

## 1) Schemas Zod (neutros, estritos e reutilizáveis)

```ts
// structured-ui.ts
import { z } from "zod";

/** Botão com título ≤ 20 chars; payload opcional */
export const Button = z.object({
  title: z.string().regex(/^.{1,20}$/u, "máx 20 caracteres"), // melhor que .max(20)
  payload: z.string().regex(/^.{0,100}$/u).nullable().default(null),
}).strict();

export const QuickReply = z.object({
  channel: z.enum(["whatsapp", "instagram", "messenger"]),
  body: z.string().regex(/^.{1,500}$/u, "máx 500 caracteres"),
  buttons: z.array(Button).min(1).max(3),
}).strict();

export type TQuickReply = z.infer<typeof QuickReply>;
export type TButton = z.infer<typeof Button>;
```

> Por que regex? Em Structured Outputs, **`pattern` é suportado de forma consistente**; já `maxLength` pode não ser aplicado em todos cenários. Assim o modelo é realmente “forçado” ao limite.

// guards.ts
export const fit = (s: string, n: number) =>
  s.length <= n ? s : s.slice(0, n).trim();

export function normalizeQuickReply(q: TQuickReply): TQuickReply {
  // clamp canal-aware (WA só aceita 1–3 botões; já garantimos no schema)
  const lim = Math.max(1, Math.min(3, q.buttons.length));
  const buttons = q.buttons.slice(0, lim).map((b, i) => ({
    title: fit(b.title, 20),
    payload: b.payload ?? `BTN_${i + 1}`,
  }));
  return {
    channel: q.channel,
    body: fit(q.body, 500),
    buttons,
  };
}

## 3) Chamada ao Responses API (preferir `instructions`)

```ts
// llm.ts
import OpenAI from "openai";
import { zodTextFormat } from "openai/helpers/zod";
import { QuickReply, TQuickReply } from "./structured-ui";

const client = new OpenAI();

const DEV_RULES = `
# OBJETIVO
Gerar uma resposta curta (body) e 2–3 botões objetivos.

# REGRAS
- Títulos de botões: até 20 caracteres (obrigatório).
- Linguagem neutra e profissional (sem emojis/hashtags).
- Se a pergunta for ambígua, use botões para esclarecer intenção.
- Retorne SOMENTE no schema 'quick_reply' (sem explicações fora do JSON).
`;

export async function generateQuickReply(args: {
  model?: string;              // "gpt-5" | "gpt-4.1"
  channel: "whatsapp" | "instagram" | "messenger";
  userText: string;
}) {
  const { model = "gpt-5", channel, userText } = args;

  const res = await client.responses.parse({
    model,
    reasoning: { effort: "low" },
    instructions: "Você é UX writer jurídico. Siga o schema estritamente.",
    input: [
      { role: "developer", content: DEV_RULES },
      { role: "developer", content: `Canal alvo: ${channel}` },
      { role: "user", content: userText },
    ],
    text: { format: zodTextFormat(QuickReply, "quick_reply") },
  });

  if (res.status === "incomplete") {
    throw new Error(`Incomplete: ${res.incomplete_details?.reason || ""}`);
  }

  if (!res.output_parsed) throw new Error("Sem output_parsed.");

  return res.output_parsed as TQuickReply;
}
```

> **Opcional**: trocar `instructions` por uma única mensagem `developer`. Em prioridade, `instructions` vence e não “polui” o histórico.

## 4) Adaptadores por canal (payload final para cada API)

```ts
// adapters.ts
import { TQuickReply } from "./structured-ui";

export function toWhatsApp(q: TQuickReply) {
  // WhatsApp interactive buttons (reply)
  return {
    type: "interactive",
    interactive: {
      type: "button",
      body: { text: q.body },
      action: {
        buttons: q.buttons.map((b) => ({
          type: "reply",
          reply: { id: b.payload ?? "BTN", title: b.title },
        })),
      },
    },
  };
}

export function toInstagram(q: TQuickReply) {
  // IG quick replies (estrutura típica de botões textuais)
  return {
    text: q.body,
    quick_replies: q.buttons.map((b) => ({
      title: b.title,
      payload: b.payload ?? "BTN",
    })),
  };
}

export function toMessenger(q: TQuickReply) {
  // Messenger quick replies
  return {
    text: q.body,
    quick_replies: q.buttons.map((b) => ({
      content_type: "text",
      title: b.title,
      payload: b.payload ?? "BTN",
    })),
  };
}

export function toChannelPayload(q: TQuickReply) {
  switch (q.channel) {
    case "whatsapp":  return toWhatsApp(q);
    case "instagram": return toInstagram(q);
    case "messenger": return toMessenger(q);
  }
}
```

## 6) Fallback automático (Structured → JSON mode + validação Zod)

```ts
// fallback.ts
import OpenAI from "openai";
import { zodTextFormat } from "openai/helpers/zod";
import { z, ZodTypeAny } from "zod";
const client = new OpenAI();

export async function structuredOrJson<T extends ZodTypeAny>(p: {
  model?: string;
  schema: T;
  name: string;
  instructions?: string;
  messages: { role: "developer" | "user" | "assistant"; content: string }[];
  max_output_tokens?: number;
}) {
  const { model = "gpt-4.1", schema, name, instructions, messages, max_output_tokens } = p;

  try {
    const r = await client.responses.parse({
      model, input: messages, instructions, max_output_tokens,
      text: { format: zodTextFormat(schema, name) },
    });
    return { mode: "structured" as const, data: r.output_parsed, raw: r };
  } catch {
    const instr = (instructions ?? "") + " Responda em JSON válido.";
    const r = await client.responses.create({
      model, input: messages, instructions: instr,
      text: { format: { type: "json_object" } }, max_output_tokens
    });
    if (r.status === "incomplete")
      throw new Error(`Incomplete: ${r.incomplete_details?.reason}`);
    const first = r.output?.[0]?.content?.[0];
    if (first?.type === "refusal") throw new Error("Refusal: " + first.refusal);

    // Validação local robusta
    const parsed = schema.parse(JSON.parse(r.output_text!));
    return { mode: "json_mode_fallback" as const, data: parsed, raw: r };
  }
}
```

## 9) Structured Outputs **vs** JSON mode (em 1 respiro)

* **Structured Outputs**: modelo **aderente ao schema** (enums/chaves), **refusal explícito**, menos retrabalho → perfeito pra **UI e contratos de dados**.
* **JSON mode**: só garante **JSON válido**; sem garantia de schema → **valide** e faça **retries**. Use como **fallback**.
